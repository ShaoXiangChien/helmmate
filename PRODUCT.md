# Product

## Register

product

## Users

HelmMate is for solo technical founders, senior engineers, technical PMs, small product teams, open-source maintainers, and consultants who use local AI coding agents inside real repositories. They are already working with branches, worktrees, tickets, acceptance criteria, logs, pull requests, and review notes, and they need a cockpit that makes agent work inspectable before it becomes risky.

The primary user context is focused engineering work: turning rough product asks into agent-ready tickets, deciding what can run, watching local execution, and reviewing handoffs. Users value precision, low surprise, and a clear trail of what happened.

## Product Purpose

HelmMate turns fuzzy engineering intent into bounded, review-gated AI coding work. It keeps agent work close to familiar software-delivery primitives: repo, branch, worktree, ticket, dependency, acceptance criteria, test command, launch log, PR handoff, and human review.

Success means a user can connect an existing repo, understand what HelmMate learned, create or sharpen a ticket, preview launch state, run a local agent only when they mean to, and review the result without wondering what changed or why.

## Brand Personality

Sharp console: dense, technical, controlled, and honest about risk. HelmMate should feel like an expert operator's panel rather than a generic productivity app.

The tone should be direct and engineering-native. It can be spare and slightly intense, but it should not become cryptic. The interface earns trust by showing state clearly, naming blockers precisely, and making unsafe transitions difficult to miss.

## Anti-references

HelmMate should not look or sound like a fully autonomous company simulator, a generic kanban clone, a cheerful no-code workflow builder, or a glossy SaaS landing page transplanted into an app shell.

Avoid claims and visuals that imply zero-human operation, agents as employees, org-chart theater, business-wide automation, or replacing engineering judgment. Avoid soft motivational copy, decorative dashboard drama, and ambiguous launch states that make queued, blocked, and running work feel interchangeable.

## Design Principles

1. Show operational truth first. WIP, launch readiness, disarmed state, blockers, logs, and review status should agree across the UI.
2. Keep autonomy human-led. Agents execute bounded tickets; the user keeps control of arming, launching, reviewing, and memory updates.
3. Make setup auditable. Existing repo setup should preview what will be read or written, preserve unrelated config, and explain validation.
4. Treat ticket quality as product quality. Context refs, dependencies, acceptance criteria, branch naming, and reviewer notes are first-class workflow objects.
5. Prefer earned density over decoration. The product can be compact and technical when the information helps decisions.

## Accessibility & Inclusion

Use practical product accessibility as a baseline: readable contrast, visible focus states, keyboard-operable controls, semantic labels for important actions, and reduced-motion alternatives for animations. The audience is technical, but the interface should still avoid low-contrast text, pointer-only workflows, hidden state, and motion that gets in the way of work.
